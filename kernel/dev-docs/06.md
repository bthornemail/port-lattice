# **ULP Lattice Runtime v1.1 — Review Submission Outline**

## 0. Executive Statement (what reviewers should understand first)

> **v1.1 enforces FIFO-first semantics and lifts the runtime into a geometric engine.**
> All communication is modeled as a bipartite incidence graph (Ports ↔ Processes).
> Network transports are projections, never primitives.
> All destructive effects are gated by a pure Coxeter analysis that computes blast radius *before* touching POSIX.

If a reviewer understands *that*, they understand the system.

---

## 1. Kernel Scope (what is considered “core”)

**Repository scope**

* `lattice-runtime` (kernel + runtime)
* No required changes to `lattice-netcat` (treated as a projection tool)

**Kernel guarantees**

* Pure compile + plan phases
* FIFO-first structural invariant
* No IO during validation, diff, or blast analysis
* POSIX effects only occur after safety checks pass

---

## 2. Core Invariants (non-negotiable)

These must be explicitly listed in the review.

### 2.1 FIFO-First Invariant

* Every `Port` **must** materialize as a FIFO
* No TCP/UDP/SSH port exists without a FIFO
* Deleting a transport never deletes a FIFO

> **Why reviewers care:** prevents accidental network authority and enables offline-first behavior.

---

### 2.2 Projection Rule

* Processes and transports **attach** to ports
* Ports are structural, transports are boundary projections
* Network failure ≠ structural failure

---

### 2.3 Idempotence

* `compile` is pure
* `plan` is deterministic
* `reconcile(plan ∘ reconcile(plan)) = reconcile(plan)`

---

## 3. Data Model (what reviewers should inspect)

### 3.1 Board (declarative intent)

* Ports (FIFO paths only)
* Processes (`waitsOn` / `firesTo`)
* Transports (attach → port)

No protocol semantics in ports.

---

### 3.2 Kernel Types (Haskell)

Reviewers should see:

* `Port`, `Process`, `Transport`
* `Board`
* `validateBoard :: Board -> [BoardError]`

Key check:

> Is it *impossible* to represent a non-FIFO port?

If yes → invariant holds.

---

## 4. Pure Geometry Layer (the “new” part in v1.1)

This is the heart of the review.

### 4.1 Bipartite Incidence Graph

* 0-cells: `Ports ∪ Processes`
* 1-cells: signed edges (`waitsOn = -1`, `firesTo = +1`)

This graph is **derived**, not authored.

---

### 4.2 Coxeter / Chain-Complex Export

Included modules:

* `extractComplex :: Board -> CoxeterComplex`
* `boundary1 :: C₁ → C₀` (sparse boundary operator)

What reviewers should verify:

* No IO
* Deterministic ordering
* Stable edge identities

---

### 4.3 Euler Characteristic

* `χ = |V| − |E|`
* Used as a fast structural sanity signal

> Not a proof — a *guardrail*.

---

## 5. Non-Destructive Safety Mechanism (review focus)

### 5.1 Pure Diff

* `diffComplex old new`
* Captures semantic change, not textual change

---

### 5.2 Patch Extraction

* Which ports/procs are touched
* Which edges are added/removed

This is the *minimal semantic patch*.

---

### 5.3 Blast Radius Analysis

* Connected components
* Affected components
* Union reachability set (“blast radius”)

**Critical rule**:

> If blast radius exceeds policy → runtime refuses to act.

No FIFO touched.
No process started.
No side effects.

---

## 6. Control Flow (end-to-end)

Reviewers should be able to trace this pipeline:

```
Board
  ↓ validateBoard (pure)
  ↓ extractComplex (pure)
  ↓ diff / blast analysis (pure)
  ↓ compile (pure)
  ↓ plan (pure)
  ↓ interpretPOSIX (IO, gated)
```

If any pure phase fails → **zero side effects**.

---

## 7. POSIX Projection (explicitly not core)

Included but clearly separated:

* `mkfifo`
* `cat`

* `ssh` (wrapper only)

Reviewers should see:

* POSIX is an *interpreter*
* Kernel does not depend on tool choice

## 8. Trace Semantics (review checkpoint)

Traces record:

* `port_materialized`
* `transport_attached`
* `process_ensured`
* `heal_attempted`

Traces **never** record:

* shell commands
* absolute paths outside declared FIFOs
* implicit authority

This preserves replayability and auditability.

---

## 9. Security & Sandbox Story (important for review)

### 9.1 Why this is safe by default

* User can only act within declared graph
* Blast radius computed before execution
* Empty / invalid boards → empty plan

### 9.2 “Topological Jail”

* If a resource isn’t a node in the graph, it is unreachable
* `/etc/passwd` cannot be touched because it is not in the complex

This is stronger than path-based sandboxing.

* No distributed consensus
* No privilege escalation
* No implicit network discovery

All decentralization happens via **declarations**, not authority.
