# RFC-0012 §X — Coxeter Safety Lens (Normative)

## X.1 Purpose

This section defines a **pure geometric safety lens** that lifts a FIFO-first lattice description into a **Coxeter system** in order to classify whether a self-described environment is **bounded** or **potentially unbounded** *before* any POSIX side effects occur.

This mechanism exists to enforce **non-destructive defaults** and provide a mathematically grounded refusal path for unsafe topologies.

---

## X.2 Scope and Applicability

This section applies to:

* `lattice-kernel` (authoritative geometry + classification)
* `lattice-runtime` (policy enforcement + execution gate)

This section does **not** apply to:

* `lattice-netcat` (transport projection only)
* any execution semantics or POSIX behavior

---

## X.3 Definitions

### X.3.1 Ports as Generators

Let:

* ( P = { p_1, p_2, \dots, p_n } ) be the set of **Ports** in a Board.

Each Port ( p_i ) SHALL be treated as a **generator** ( s_i ) of a Coxeter system.

Processes SHALL NOT be treated as generators.

---

### X.3.2 Derived Port Co-Occurrence

Given the bipartite incidence graph:

* Ports ↔ Processes
  defined by `waitsOn` and `firesTo`

Define the **co-occurrence count** ( c_{ij} ) between ports ( p_i ) and ( p_j ) as:

> The number of *distinct processes* in which both ( p_i ) and ( p_j ) appear (in either `waitsOn` or `firesTo`).

This count MUST be symmetric and MUST ignore multiplicity within a single process.

---

## X.4 Coxeter Matrix Construction (Normative)

From the co-occurrence counts, construct the **Coxeter matrix** ( M = (m_{ij}) ) as follows:

* ( m_{ii} = 1 )
* If ( c_{ij} = 0 ):
  ( m_{ij} = 2 ) (orthogonal generators)
* If ( c_{ij} = 1 ):
  ( m_{ij} = 3 )
* If ( c_{ij} \ge 2 ):
  ( m_{ij} = 4 )

> Implementations MAY cap ( m_{ij} ) at a higher value (e.g. 6),
> but MUST NOT assign values less than 2 for ( i \neq j ).

This rule defines the **unique Coxeter diagram** associated with a Board under this RFC.

---

## X.5 Gram Matrix and Classification

From the Coxeter matrix ( M ), construct the **Gram matrix** ( G ):

* ( G_{ii} = 1 )
* ( G_{ij} = -\cos(\pi / m_{ij}) ) for ( i \neq j )

The Coxeter system SHALL be classified as:

* **Finite**
  if ( G ) is positive definite
* **Affine-Like**
  if ( G ) is positive semidefinite but not positive definite
* **Indefinite**
  otherwise

This classification MUST be computed using a deterministic, pure algorithm (e.g. Cholesky decomposition).

---

## X.6 Execution Gating Rules (Normative)

### X.6.1 Kernel Output

The kernel SHALL emit:

* ordered port list
* Coxeter matrix ( M )
* Gram matrix ( G )
* classification: `finite | affine_like | indefinite`

No POSIX actions MAY occur during this phase.

---

### X.6.2 Runtime Policy

The runtime MUST apply policy **after** kernel classification and **before** reconcile.

The default policy SHALL be:

* `finite` → **ACCEPT**
* `affine_like` → **REFUSE**
* `indefinite` → **REFUSE**

A runtime MAY allow overrides via explicit policy configuration, but refusal MUST remain the default.

---

### X.6.3 Refusal Semantics

If execution is refused due to Coxeter classification:

* No FIFOs SHALL be created
* No transports SHALL be attached
* No processes SHALL be spawned
* A trace event MUST be recorded:

  * `action = blast_refused`
  * `reason = coxeter_unbounded`

This preserves the **non-destructive default**.

---

## X.7 Relationship to Blast-Radius Analysis

The Coxeter Safety Lens complements (but does not replace):

* diff-based blast-radius analysis
* connected-component analysis

The Coxeter classification answers:

> *Is the topology itself bounded under reflection?*

Blast-radius answers:

> *How much existing structure would this change affect?*

Both MUST pass for execution to proceed.

---

## X.8 Wythoff Construction (Non-Normative)

Given a **Finite** Coxeter classification, implementations MAY:

* choose a seed point in the fundamental simplex
* enumerate vertices via reflections
* use this polytope for:

  * visualization
  * state-space reasoning
  * documentation

Full Wythoff enumeration is OPTIONAL and SHALL NOT be required for execution gating in v1.2.

---

## X.9 Rationale

This mechanism ensures that:

* self-described environments are mathematically constrained
* unbounded or runaway topologies are refused *by construction*
* safety derives from geometry, not ad-hoc heuristics
* decentralization does not introduce unbounded execution risk

---

## X.10 Summary (Normative)

> A lattice implementation conforming to RFC-0012 MUST refuse to execute any Board whose derived Coxeter system is not finite, unless an explicit policy override is present.
