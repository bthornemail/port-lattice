# RFC-0012: Decentralized Declarations & Gossip for the Lattice

**Version:** v1.2 (Draft)
**Status:** Proposal
**Scope:** Decentralization (Discovery, Membership, Topology Convergence)
**Applies to:** lattice-kernel, lattice-runtime, lattice-netcat (compatibility only)

---

## 1. Abstract

This RFC specifies a **decentralized declaration and gossip layer** for the Lattice ecosystem.

Rather than distributing executable configuration or control authority, peers exchange **signed declarations of participation**. Each node independently resolves these declarations into a **derived local board**, which is then processed by the existing FIFO-first lattice pipeline.

This preserves:

* local determinism
* trace sovereignty
* non-destructive defaults
* offline-first behavior

No global coordinator, consensus protocol, or privileged control plane is introduced.

---

## 2. Non-Goals (Explicit)

This RFC does **not** define:

* distributed consensus
* leader election
* shared global state
* automatic execution of remote code
* mandatory transport or network stack

Decentralization here means **independent convergence**, not shared authority.

---

## 3. Architectural Principle

> **Declarations are data, not commands.**
> **Resolution is local, deterministic, and auditable.**

Every node remains sovereign over:

* which declarations it accepts
* how they are resolved
* whether they are executed at all

---

## 4. Layering Model (Compatibility Guarantee)

This RFC introduces **no coupling** between projects.

### 4.1 lattice-kernel (unchanged)

* Defines:

  * FIFO-first ports
  * processes
  * transports
  * Coxeter geometry
  * blast-radius analysis
* Adds:

  * **no networking**
  * **no gossip**
  * **no identity layer**

Kernel remains **pure**.

---

### 4.2 lattice-runtime (extended, optional)

* Gains:

  * declaration ingestion
  * resolver invocation
  * trace recording of declaration events
* Does **not**:

  * enforce decentralization
  * require networking

Runtime remains usable in fully local mode.

---

### 4.3 lattice-netcat (unchanged)

* Used only as a **transport projection**
* May be used to:

  * carry gossip payloads
  * bridge declaration channels
* Has **no awareness** of declarations or lattice semantics

---

## 5. Declaration Model

### 5.1 Declaration Ledger

Declarations are append-only events, typically stored as:

```
decl.jsonl
```

Each line is a **signed declaration event**.

---

### 5.2 Declaration Schema (Normative)

```json
{
  "decl_id": "sha256:…",
  "type": "peer_announce | service_offer | transport_offer | revoke",
  "peer_id": "peer:ed25519:…",
  "timestamp": 1735689600,
  "payload": { },
  "signature": "ed25519:…"
}
```

#### Required properties

* MUST be immutable
* MUST be verifiable
* MUST be replayable
* MUST NOT contain executable instructions

---

### 5.3 Declaration Types

#### 5.3.1 `peer_announce`

Declares existence and identity.

Payload example:

```json
{
  "display_name": "edge-node-7",
  "pubkey": "ed25519:…"
}
```

---

#### 5.3.2 `service_offer`

Declares a logical service (no transport).

```json
{
  "service": "metrics",
  "interface": "line-protocol",
  "direction": "in"
}
```

---

#### 5.3.3 `transport_offer`

Declares *possible* connectivity.

```json
{
  "service": "metrics",
  "protocol": "tcp",
  "endpoint": "tcp://203.0.113.5:8125"
}
```

This does **not** create a port.
It is an *option* that may be ignored.

---

#### 5.3.4 `revoke`

Withdraws a prior declaration by ID.

---

## 6. Gossip / Distribution Layer

### 6.1 Transport-Agnostic

This RFC deliberately does **not** mandate a gossip protocol.

Valid options include:

* MQTT (brokered or federated)
* libp2p pubsub
* LAN broadcast
* file sync
* sneakernet

All transports are **out of scope** and treated as projections.

---

### 6.2 Gossip Semantics

* Declarations MAY be duplicated
* Order MUST NOT matter
* Loss MUST be tolerated
* Replay MUST be safe

---

## 7. Resolver (Critical Component)

Each node runs a **resolver**:

```
(local template board)
+ (accepted declarations)
= derived board
```

### 7.1 Resolver Properties

* Pure
* Deterministic
* Side-effect free
* Policy-driven

---

### 7.2 Output

Resolver emits:

* `ports.d/*.json`
* `transports.d/*.json`
* `procs.d/*.json`

These are then fed to `lattice-runtime`.

---

### 7.3 Failure Behavior

If resolution fails:

* no FIFOs created
* no transports attached
* trace records refusal

This preserves the **non-destructive default**.

---

## 8. Trace Integration

New trace events (normative):

* `decl_received`
* `decl_verified`
* `decl_rejected`
* `board_derived`
* `board_refused`

Traces record **why** topology changed, not just *that* it did.

---

## 9. Geometry & Safety

Before execution:

1. Derived board → Coxeter complex
2. Diff vs previous complex
3. Blast radius computed
4. Policy applied

Execution proceeds **only if allowed**.

This makes decentralization **safer than local config editing**.

---

## 10. Security Model

### 10.1 Identity

* Cryptographic identity only
* No implicit trust
* No transitive authority

---

### 10.2 Containment

* Declarations cannot escape declared topology
* Resolver cannot introduce paths outside policy
* POSIX access limited to FIFO graph

This is a **topological sandbox**, not a permission list.

---

## 11. Compatibility & Migration

* v1.1 boards remain valid
* Decentralization is opt-in
* Resolver can be disabled entirely
* Kernel remains unchanged

---

## 12. Reference Flow (Illustrative)

```
peer announces → declaration received
             → declaration accepted
             → resolver derives board
             → Coxeter diff + blast check
             → reconcile executes (or refuses)
```

No remote peer ever executes code on another peer.

---

## 13. Rationale (Why this works)

This design:

* avoids consensus complexity
* preserves local sovereignty
* enables offline-first operation
* keeps projects separable
* allows competing implementations to coexist

Decentralization emerges from **shared structure**, not shared control.

---

## 14. Future Work (Non-Normative)

* Declaration compression / Merkleization
* Trust lenses (web-of-trust overlays)
* Multiple resolvers per node
* Visualization of multi-peer Coxeter complexes

---

## 15. One-Line Summary (for reviewers)

> **v1.2 introduces decentralized declarations and gossip without introducing shared authority, coupling, or non-determinism, preserving FIFO-first semantics and geometric safety guarantees.**
